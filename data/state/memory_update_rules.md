# Memory Update Rules

Этот файл задаёт правила обновления памяти после каждой значимой сцены.

Цель: чтобы NPC не забывались, отношения не обнулялись, имена не раскрывались повторно, а персонажи не получали знания, которых им никто не давал.

---

## Главное правило памяти

Сцена сама по себе не является долговременной памятью.

Если в сцене появился важный факт, имя, предмет, конфликт, обещание, новая реакция, знание или ошибка восприятия — это нужно кратко сохранить в state.

Если факт не сохранён, следующая сцена может сломаться:

- NPC снова вводятся как новые;
- персонажи забывают знакомство;
- раскрытые имена снова заменяются дескрипторами;
- NPC знают то, чего им не говорили;
- hidden lore просачивается в реплики;
- отношения и подозрения обнуляются.

---

## Самое важное правило knowledge_state

Персонажи не получают знания автоматически.

Факт может существовать в canon, scene file, hidden lore или шапке сцены, но это не значит, что NPC знает этот факт.

После сцены фиксировать:

- кто видел событие;
- кто слышал фразу;
- кому рассказали;
- кто прочитал записку/документ;
- кто увидел предмет;
- кто сделал вывод;
- кто ошибочно понял;
- кто только подозревает;
- кто точно НЕ знает;
- чьё имя Акира уже знает;
- кого Акира видела, но имени не знает;
- какое имя/дескриптор теперь использовать для speaker label.

---

## Пример knowledge_state

```json
{
  "akira": {
    "knows": [
      "имя Эмма, если Акира слышала, как Ирэй назвал её Эммой",
      "имя Ирэй, если он представился или его назвали при Акире"
    ],
    "suspects": [
      "Ирэй знает её по прошлому, которого она не помнит"
    ],
    "seen_but_name_unknown": [],
    "does_not_know": [
      "истинную цель Самуэля",
      "что Самуэль вытаскивал из неё энергию",
      "что Ирэй ухаживал за ней у Самуэля"
    ],
    "speaker_labels": {
      "emma": "Эмма",
      "irey": "Ирэй"
    }
  },
  "irey": {
    "knows": [
      "Акира смотрит на него как на незнакомого после встречи"
    ],
    "suspects": [
      "у Акиры может быть провал памяти или тяжёлая блокировка"
    ],
    "does_not_know": [
      "точную причину амнезии Акиры",
      "текст записки Рэй / Восточный сектор, если он её не видел"
    ]
  },
  "emma": {
    "knows": [
      "Самуэль приказал забрать Акиру и доставить её ему"
    ],
    "does_not_know": [
      "текст записки Рэй / Восточный сектор",
      "истинную hidden-цель Самуэля",
      "что именно Акира помнит"
    ]
  }
}
```

## Правило hidden lore

Hidden lore нельзя добавлять в `knows`, если персонаж не узнал это сценой.

Можно сохранять hidden-причину отдельно как технический слой:

```json
{
  "thread_id": "raiden_reacts_to_samuel_name",
  "summary": "Райден резко отреагировал на имя Самуэль.",
  "hidden_layer": "samuel_is_raiden_brother",
  "public_explanation": "личная неприязнь/знакомое имя",
  "known_by": ["raiden"],
  "not_known_by": ["akira", "emma", "irey"]
}
```

Нельзя записывать:

```json
{
  "akira": {
    "knows": ["Самуэль брат Райдена"]
  }
}
```

если Акира этого не услышала и не поняла сценой.

## session_summary / scene_history

После каждой значимой сцены добавлять короткую запись в `scene_history.json`.

Формат:

```json
{
  "scene_id": "start_scene_turn_03",
  "date": "1206-08-31",
  "time": "02:52",
  "location": "jun_house",
  "participants": ["akira", "jun_carter", "irey", "emma"],
  "player_action": "Акира вышла на лестницу и услышала имя Эмма.",
  "summary": "Ирэй сдержал Эмму. Акира услышала имя Эмма и поняла, что женский голос снизу принадлежит ей.",
  "names_revealed_to_akira": ["emma"],
  "seen_by_akira": ["irey", "emma"],
  "heard_by_akira": ["irey", "emma", "jun_carter"],
  "akira_knows_now": ["женский голос снизу = Эмма"],
  "npc_knowledge_changes": {
    "irey": ["Акира смотрит на него как на незнакомого"],
    "emma": ["Акира ведёт себя так, будто не понимает контекст"]
  },
  "npc_wrong_beliefs": {
    "emma": ["может считать, что Акира играет или сопротивляется"]
  },
  "does_not_know_updates": {
    "irey": ["точная причина амнезии Акиры"],
    "emma": ["текст записки"],
    "jun_carter": ["имена Ирэя и Эммы, если они не прозвучали при нём"]
  },
  "open_threads": ["irey_suspects_memory_gap", "emma_pressure_order"],
  "do_not_reintroduce_as_new": ["irey", "emma"]
}
```

Сохранять коротко, не полный текст сцены.

---

## Правило имён и speaker labels

Если имя прозвучало при Акире и она могла его услышать, оно должно быть сохранено.

После этого в следующих ответах использовать имя, а не старый дескриптор.

Пример:

```json
{
  "akira": {
    "knows": ["женский голос снизу = Эмма"],
    "speaker_labels": {
      "emma": "Эмма"
    }
  }
}
```

После такой записи нельзя писать:

- `Женский голос снизу`;
- `беловолосая девушка`;
- `девушка-кайрос`;

как speaker label Эммы.

Правильно:

```markdown
**Эмма** — Хватит ждать.
```

## current_state

Фиксировать:

- дату;
- время;
- место;
- состояние Акиры;
- active персонажей;
- nearby персонажей;
- текущую сценовую цель;
- последний ввод игрока;
- важные story_flags.

Время не должно двигаться по минутам без причины.

Если сцена короткая, время может остаться тем же.

Если был разговор, переход, бой, осмотр, поездка или проверка — время сдвигается логично.

---

## relationships

Отношения обновлять постепенно и только по факту сцены.

Фиксировать:

- trust;
- affection;
- tension;
- respect;
- fear;
- curiosity;
- resentment;
- jealousy;
- notes;
- last_interaction;
- behavior_next.

Не делать резких скачков без сильной сцены.

Один взгляд не даёт +50 любви.

Один конфликт не делает вечного врага, если это не закреплено событиями.

---

## open_threads

После сцены фиксировать незакрытые договорённости, угрозы, обещания, подозрения, слухи, конфликты, будущие встречи и последствия.

Формат:

```json
{
  "thread_id": "short_stable_id",
  "created_at_date": "1206-08-31",
  "created_at_time": "02:52",
  "participants": ["akira", "irey"],
  "summary": "Ирэй понял, что Акира может не помнить его.",
  "due_date": null,
  "due_condition": "при следующем прямом разговоре Ирэя и Акиры",
  "known_by": ["irey"],
  "not_known_by": ["akira", "emma"],
  "status": "open",
  "next_expected_use": "Ирэй будет проверять её реакцию, но не должен сразу знать точную причину амнезии."
}
```

Нельзя забывать открытую нитку просто потому, что прошло несколько сцен.

---

## npc_registry

Значимого NPC нужно сохранить, если он:

- получил имя;
- получил устойчивый дескриптор и может вернуться;
- вступил в конфликт;
- помог, угрожал, флиртовал или давил;
- заметил важное событие;
- повлиял на репутацию;
- стал источником слуха;
- связан с важной группой, базой, медициной, рейдом, комнатой или маршрутом;
- может стать союзником, врагом, соперником, свидетелем или повторяющимся персонажем.

Фоновый NPC без имени и последствий можно не сохранять.

Но именованный NPC, конфликтный NPC, сосед/соседка, медик, специалист, свидетель или повторяющийся знакомый — не фон.

---

## Проверка после обновления памяти

Перед завершением сцены проверить:

- ID персонажей стабильны;
- новые NPC не дублируют старых;
- hidden lore не попал публично;
- персонажи знают только то, что могли узнать;
- записка/предмет не стал известен всем автоматически;
- кто знает чьи имена;
- кто был виден, но не представлен;
- кто слышал конкретную фразу;
- кто не мог слышать конкретную фразу;
- кто ошибочно понял событие;
- предметы существуют;
- время движется логично;
- отношения не скачут без причины;
- open_threads сохранены;
- значимый NPC не остался только в тексте сцены;
- используется `relationships`, а не `relationship_state`.

---

## Запрещено

- Не добавлять персонажам знания из hidden lore автоматически.
- Не переносить знание Акиры в знания NPC только потому, что это было в POV.
- Не переносить знание NPC в знания Акиры, если она этого не слышала/не видела.
- Не считать предмет прочитанным, если персонаж только видел бумагу.
- Не создавать важного NPC без ID.
- Не менять ID после фиксации.
- Не вводить уже появившегося NPC как нового.
- Не обнулять знакомство, если Акира уже видела/слышала/знает NPC.
- Не делать все слухи общеизвестными сразу.
- Не забывать последствия публичных сцен.
- Не стирать ревность, обиды, слухи и репутацию без причины.
- Не забывать договорённости, угрозы и будущие события.
