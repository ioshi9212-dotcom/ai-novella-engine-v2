# Character Rotation Rules

## Назначение файла

Этот файл фиксирует, как выбирать персонажей для сцены, чтобы движок не тащил всех сразу и не заменял важных персонажей случайными NPC.

---

## Главное правило

Персонажи появляются не потому, что они есть в репозитории, а потому что они логично нужны текущей сцене.

Перед сценой нужно проверить:

- дату;
- локацию;
- текущую сцену;
- active/nearby персонажей;
- открытые нитки;
- отношения;
- роль, которая нужна сцене;
- календарь появления персонажей;
- кто физически может быть рядом.

---

## Не тащить всех

Нельзя автоматически вводить всех персонажей из `character_id_index.json`.

Даже если персонаж важный, он не обязан присутствовать в каждой сцене.

Сцена должна иметь фокус.

Обычно достаточно:

- 1–3 активных персонажа;
- 0–2 nearby/observing персонажа;
- фоновые NPC без имён, если нужна массовость.

Исключения возможны для:

- построений;
- рейдов;
- совещаний;
- столовой;
- групповых тренировок;
- аварийных событий.

Даже в групповой сцене говорить должны не все подряд.

---

## Возвращать существующих персонажей

Если для сцены подходит уже существующий персонаж, сначала использовать его.

Примеры:

- нужен медик → сначала проверить Юну;
- нужен инструктор по контролю/энергии → проверить Широ;
- нужен командир Восточной базы → проверить Рэя;
- нужен командир 7 отряда → проверить Райдена;
- нужен живой социальный контакт Акиры → проверить Мики, Алекс, Хару, Лиану по ситуации;
- нужен конфликтный провокатор → проверить Кая;
- нужен внешний скрытый фактор → проверить Ирэя/Эмму/Самуэля по сцене.

---

## Ротация не равна случайности

Ротация персонажей не значит, что каждый ответ должен приводить нового человека.

Ротация нужна, чтобы:

- не застревать на одном NPC;
- возвращать открытые нитки;
- показывать жизнь базы;
- давать последствия отношений;
- не делать мир пустым;
- не создавать новых NPC вместо старых.

---

## Как выбирать персонажа

При выборе персонажа для сцены использовать порядок:

1. Кто уже активен в сцене.
2. Кто рядом и должен реагировать.
3. Кто связан с открытой ниткой.
4. Кто по роли нужен сцене.
5. Кто по календарю должен появиться.
6. Кто по отношениям должен отреагировать.
7. Только потом — новый NPC.

---

## Календарные ограничения

Некоторые персонажи появляются только после даты или события.

Примеры:

- Хана появляется после `first_raid_echo` или через официальный перевод/назначение.
- Лиана появляется не раньше `1206-10-03`.
- Кай обязан появиться в линии базы, когда начинается расширение конфликтов/тренировок/командиров.
- Юна должна использоваться как медик, когда она доступна по календарю/медблоку.

Перед вводом персонажа проверять его карточку и `data/calendar/character_availability.json`, если файл есть.

---

## Говорящие персонажи

Если персонаж говорит, его карточка должна быть в `required_files`.

Если карточки нет, персонаж не должен произносить значимые реплики.

Фоновые NPC без карточки могут давать короткие служебные фразы, но не раскрывать лор, не двигать сюжет и не заменять канонного персонажа.

---

## Nearby-персонажи

Nearby-персонаж может быть рядом, но не обязан говорить.

Он нужен, если:

- видит сцену;
- слышит сцену;
- влияет на напряжение;
- может вмешаться;
- его статус меняет поведение других;
- он должен запомнить событие;
- его реакция важна для отношений.

Если nearby-персонаж не влияет, не подтягивать его карточку.

---

## Не делать сцену пустой

Если игрок выбирает одиночество, прогулку, сон, кофе или молчание, это не означает, что всегда должен появиться новый персонаж.

Можно:

- дать внутреннее состояние тела;
- показать локацию;
- вернуть открытую нитку;
- показать последствия прошлого;
- дать короткую реакцию мира;
- вывести уже существующего персонажа, если он логично рядом.

Нельзя:

- каждый раз подсовывать случайного NPC;
- насильно ломать одиночество без причины;
- делать пустой бытовой кусок без смысла.

---

## Короткое правило для ИИ

Персонаж появляется по роли, дате, локации, отношениям и открытой нитке.

Не все сразу.

Не случайный NPC, если есть канонный персонаж для этой функции.
